<wsl2_architecture_context>
<!-- CRITICAL DECISION CONTEXT — read before implementing -->

The user runs Chrome on Windows, dev tools in WSL2. Chrome on Windows CANNOT see Linux virtual devices
(guaranteed — Chrome uses Windows DirectShow/WASAPI, not Linux /dev/videoN or PulseAudio).

Two possible paths:

PATH A — Chrome inside WSL2 via WSLg:
  Requires: Custom WSL2 kernel compiled with v4l2loopback (CONFIG_V4L2_LOOPBACK=m)
  Audio: WSLg uses PulseAudio over RDP — pactl null-sink modules may disappear after ~2 min (known bug)
  Status: High effort, uncertain stability

PATH B — Chrome on Windows + Windows-side bridges (RECOMMENDED FALLBACK):
  Camera: OBS Studio + OBS Virtual Camera plugin (Windows DirectShow device Chrome sees natively)
  Audio:  VB-Cable (Windows virtual audio cable, Chrome sees it as a real Windows mic)
  Node.js audio bridge: Write audio from WSL2 to VB-Cable via PulseAudio TCP → Windows PulseAudio
  Status: More steps, but each component is battle-tested

The CONTEXT.md locked decision: "If WSL2 virtual devices prove too difficult, document limitations and
ship native Linux support — don't block the project." The fallback (VB-Cable + OBS) is acceptable.

Implementation strategy:
1. Detect WSL2 version with `wsl --version` (from inside WSL2 via `wsl.exe --version`)
2. Probe the WSLg path: try loading v4l2loopback via modprobe — if it works, test camera
3. If modprobe fails (DKMS can't compile for WSL2 kernel), declare PATH B and document it
4. For audio: test `pactl list sources` and check for WSLg stability
5. Document whatever works with exact steps
</wsl2_architecture_context>

**Step 1: Probe the WSL2 environment (run these commands manually to gather data):**

```bash
# Check WSL version
wsl.exe --version 2>/dev/null || echo "wsl.exe not reachable from inside WSL2"

# Check if v4l2loopback DKMS compiled for this kernel
uname -r
lsmod | grep v4l2loopback  # If nothing, DKMS didn't compile it

# Attempt to load v4l2loopback (this may fail on WSL2)
sudo modprobe v4l2loopback video_nr=10 card_label="AI Meet Agent Camera" exclusive_caps=1 max_buffers=2 2>&1

# After attempt, check if device was created
ls /dev/video10 2>/dev/null && echo "Camera device created" || echo "Camera device NOT created"

# Check PulseAudio/PipeWire state under WSLg
pactl info 2>&1 | head -5
pactl list sources short 2>&1 | head -10

# Load a test audio module and check stability
TEST_MODULE=$(pactl load-module module-null-sink sink_name=wsl2_test 2>&1)
echo "Module ID: $TEST_MODULE"
sleep 10
pactl list modules short | grep wsl2_test && echo "Module still present" || echo "Module disappeared (WSLg eviction bug confirmed)"
pactl unload-module $TEST_MODULE 2>/dev/null || true
```

- If `modprobe v4l2loopback` succeeded AND `/dev/video10` exists → PATH A (WSLg) may work for camera. Continue probing audio stability.
- If `modprobe v4l2loopback` failed → Declare PATH B (Windows bridges). Do NOT attempt kernel compilation — CONTEXT.md locked decision says don't block the project.
- If audio module disappears after 10 seconds → PATH A audio is unreliable. Document PATH B for audio even if PATH A camera works.

**Step 3: Create src/platform/wsl2.ts:**

```typescript
import { execSync } from 'child_process';

export type Wsl2Path = 'wslg' | 'windows-bridge' | 'unknown';

export interface Wsl2Status {
  path: Wsl2Path;
  checks: { name: string; ok: boolean; note?: string }[];
}

/**
 * Probe the WSL2 environment to determine which device path is viable.
 * Does NOT create any devices — only checks what's available.
 */
export function checkWsl2Prerequisites(): Wsl2Status {
  const checks: { name: string; ok: boolean; note?: string }[] = [];

  // Can v4l2loopback module be loaded?
  let cameraPath: Wsl2Path = 'unknown';
  try {
    execSync('lsmod | grep -q v4l2loopback', { stdio: 'pipe' });
    checks.push({ name: 'v4l2loopback module loaded', ok: true, note: 'WSLg camera path may work' });
    cameraPath = 'wslg';
  } catch {
    checks.push({
      name: 'v4l2loopback module loaded',
      ok: false,
      note: 'DKMS module not available for WSL2 kernel — use Windows-side OBS Virtual Camera',
    });
    cameraPath = 'windows-bridge';
  }

  // Is pactl responsive?
  try {
    execSync('pactl info', { stdio: 'pipe' });
    checks.push({ name: 'PulseAudio (pactl)', ok: true, note: 'Audio may work under WSLg (stability: uncertain)' });
  } catch {
    checks.push({
      name: 'PulseAudio (pactl)',
      ok: false,
      note: 'pactl not available — use VB-Cable on Windows side',
    });
    cameraPath = 'windows-bridge';
  }

  return {
    path: cameraPath,
    checks,
  };
}
```
