**Change A: Add import** (at top of file with other imports):

**Change B: Start relay BEFORE audio pipeline** — Insert a new block between the WSL2 status message (around line 48) and the audio pipeline creation (line 51 `let capture`). The relay must be started and listening BEFORE `createAudioCapture()` and `createAudioOutput()` are called, because those functions immediately try to connect to the relay.

```typescript
// Start WSL2 audio relay (must be before audio pipeline creation)
let relayServer: WslAudioRelayServer | null = null;
if (platform === 'wsl2') {
  try {
    relayServer = new WslAudioRelayServer(config);
    await relayServer.start();
    console.log(`[AudioRelay] TCP relay listening on port ${RELAY_PORT}`);
  } catch (err) {
    console.warn(`[AudioRelay] Could not start relay: ${(err as Error).message}`);
    console.warn('[AudioRelay] WSL2 audio pipeline will not be active.');
  }
}
```

**Change C: Add relay to shutdown handler** — In the `shutdown` function, stop the relay server AFTER stopping audio clients but BEFORE `manager.shutdown()`. The relay must outlive the audio clients so they can disconnect cleanly.

```typescript
// In shutdown(), after output.stop() and before videoFeed.stop():
if (relayServer) {
  try { relayServer.stop(); } catch { /* ignore */ }
}
```

The shutdown order becomes: session → capture → output → relayServer → videoFeed → manager

**Important:** Do NOT change the non-fatal try/catch pattern around the audio pipeline creation (lines 56-83). The relay is independently non-fatal — if it fails to start, the audio pipeline creation will still be attempted (and will likely fail with ECONNREFUSED on WSL2, which is handled by the existing non-fatal pattern).

**1. Update `scripts/setup-wsl2-windows.md`** — Add a comprehensive "Audio Relay Setup" section to the existing WSL2 setup guide. If the file does not exist, create it. The section should cover:

**VB-Cable Installation:**
- Download from https://vb-audio.com/Cable/
- Run VBCABLE_Setup_x64.exe as administrator
- Reboot after installation
- Verify: "CABLE Input" and "CABLE Output" appear in Windows Sound settings

**Chrome Audio Routing:**
- In Google Meet, click ⋮ → Settings → Audio
- Set "Speakers" to "CABLE Input (VB-Audio Virtual Cable)"
- This routes Meet participant audio through VB-Cable so the relay can capture it
- Alternative: Set CABLE Input as system default (affects all apps)

**Operator Monitoring (hearing Meet audio through speakers):**
- Open Windows Sound settings → Recording devices
- Right-click "CABLE Output (VB-Audio Virtual Cable)" → Properties
- Go to "Listen" tab → check "Listen to this device"
- Select operator's speakers/headphones as playback device → OK
- This mirrors CABLE Output to speakers while still allowing the relay to capture it

**ffmpeg/ffplay Installation:**
- Install via winget: `winget install Gyan.FFmpeg`
- Or download from https://www.gyan.dev/ffmpeg/builds/ (essentials build)
- Verify from WSL2: `ffmpeg.exe -version`
- If not on PATH, set `wsl2.ffmpegPath` and `wsl2.ffplayPath` in config.json to the full Windows path (e.g., `/mnt/c/Users/chris/AppData/Local/Microsoft/WinGet/Links/ffmpeg.exe`)

**Device Verification:**
- List capture (input) devices: `ffmpeg.exe -list_devices true -f dshow -i dummy`
- Look for "CABLE Output (VB-Audio Virtual Cable)" — this is the capture device
- If the name differs, set `wsl2.captureDevice` in config.json
- Find output device index: run `ffplay.exe -f s16le -ar 16000 -ac 1 -nodisp -i /dev/zero` — if wrong device plays, try different `-audio_device_index` values
- Set `wsl2.outputDeviceIndex` in config.json to the correct index for CABLE Input

**Config Example:**
```json
{
  "wsl2": {
    "captureDevice": "CABLE Output (VB-Audio Virtual Cable)",
    "outputDeviceIndex": 2,
    "ffmpegPath": "ffmpeg.exe",
    "ffplayPath": "ffplay.exe"
  }
}
```

- Should see: `[AudioRelay] Capture client connected` and `[AudioRelay] Output client connected`
